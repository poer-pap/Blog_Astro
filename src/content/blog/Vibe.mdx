# Vibe Coding：AI 时代的新编程方式

Vibe Coding 不是简单地让 AI 帮你写几行代码，也不是完全不懂技术就把项目交给 AI。它更像是一种新的编程协作方式：人负责表达目标、判断方向和验证结果，AI 负责快速生成代码、补全实现和辅助调试。

在传统编程里，开发者通常需要先想清楚具体实现，再一行一行写代码。而 Vibe Coding 的重点变成了：先描述你想要的效果，让 AI 给出第一版实现，然后通过运行、反馈、修改，不断把项目调整到可用状态。

简单来说，传统编程更像是：

```text
我知道怎么实现 → 我写代码 → 我调试 → 我交付
```

而 Vibe Coding 更像是：

```text
我知道想要什么 → 我描述目标 → AI 生成方案 → 我运行验证 → 我继续调整
```

这也是它最核心的变化：编程的重心从“手工写代码”转移到了“表达需求、控制方向、验证结果”。

## 一、从代码优先变成意图优先

Vibe Coding 的第一步，不是马上思考函数、组件、接口怎么写，而是先把需求讲清楚。

比如你想做一个个人导航站，如果只对 AI 说：

```text
帮我写一个首页。
```

这个提示太模糊，AI 很容易自由发挥，生成出来的东西可能和你想要的不一样。

更好的表达方式是：

```text
我要做一个个人导航站首页。左侧是分类导航，右侧是网站卡片列表。支持搜索、暗色模式和移动端自适应。整体风格要简洁，像个人工具箱页面。数据先写死，后续可以抽成配置文件。
```

这段描述里面包含了目标、页面结构、功能、风格和后续扩展方向。AI 得到的信息越清楚，生成结果就越接近你的预期。

所以，Vibe Coding 的关键能力不是会不会写某个语法，而是能不能把想法准确地表达出来。

## 二、先做出原型，再逐步完善

Vibe Coding 很适合快速做原型。你不需要一开始就把架构设计得非常复杂，可以先让 AI 做一个能跑起来的版本。

比较合理的节奏是：

```text
第一步：先跑通基础页面
第二步：补充主要功能
第三步：调整样式和交互
第四步：整理代码结构
第五步：补测试、错误处理和安全检查
```

很多人使用 AI 编程失败，是因为一开始就让 AI 做一个完整系统，例如：

```text
帮我做一个完整的 SaaS 网站，包含登录、支付、后台、数据库和部署。
```

这种需求太大，AI 很容易遗漏细节，也很难一次性生成稳定结果。

更好的方式是把任务拆小：

```text
先完成首页布局。
然后增加导航栏。
再把卡片数据抽离成配置文件。
接着加入搜索功能。
最后检查构建是否通过。
```

小步修改的好处是：每一步都能验证，每一步都容易回滚。如果出了问题，也更容易定位是哪次修改造成的。

## 三、人负责判断，AI 负责生成

Vibe Coding 并不意味着把所有决策都交给 AI。更准确地说，AI 是一个执行很快的助手，但最终判断仍然要由人来完成。

人需要负责这些事情：

```text
产品方向是否正确
功能是否符合需求
交互体验是否合理
代码是否容易维护
是否存在安全风险
最终结果是否可以交付
```

AI 更适合负责这些事情：

```text
生成初版代码
补全样板代码
修复常见报错
解释已有代码
重构局部文件
生成测试用例
整理文档说明
```

如果只是让 AI 写完代码就直接使用，而自己完全不看、不跑、不理解，这种方式风险很高。尤其是涉及用户数据、登录权限、支付接口、API Key 的项目，更不能只靠“看起来能运行”。

真正有效的 Vibe Coding，是人和 AI 分工明确：AI 提高速度，人负责质量。

## 四、用反馈循环代替一次性提示词

Vibe Coding 的核心工作流不是“一句话生成完整项目”，而是不断反馈和迭代。

一个更实际的流程是：

```text
提出目标 → AI 生成代码 → 本地运行 → 发现问题 → 反馈给 AI → 修改代码 → 再次验证
```

例如你运行项目后出现报错，不要只说“报错了”，而是把报错信息、运行命令、相关文件一起告诉 AI。

更好的反馈方式是：

```text
我执行 bun run build 后报错，错误信息如下：

这里粘贴完整报错。

请先分析原因，不要直接大改项目。只修改必要文件，并告诉我如何验证。
```

这样 AI 才能基于真实信息定位问题，而不是凭空猜测。

## 五、用验收标准控制结果

很多时候，AI 生成的代码看起来可以，但并不一定真的符合你的需求。为了避免这种情况，最好在提示词里加入明确的验收标准。

例如：

```text
完成后必须满足：
1. bun run build 可以通过
2. 页面在手机宽度下不横向滚动
3. 搜索功能可以按标题和描述过滤
4. 不引入新的大型依赖
5. 不破坏现有暗色模式
```

有了这些标准，AI 的修改范围会更受约束，你自己也更容易判断任务是否完成。

对于代码项目，尤其建议每次都加上这几句话：

```text
不要重写整个项目。
只修改和当前任务相关的文件。
修改前先说明计划。
修改后告诉我改了哪些文件，以及如何验证。
```

这能明显减少 AI 乱改文件、过度重构、引入新问题的概率。

## 六、适合和不适合的场景

Vibe Coding 很适合下面这些场景：

```text
个人博客
静态网站
小工具
Demo 原型
内部脚本
学习项目
低风险自动化
```

比如用 Astro 做一个个人站，用 AI 帮你改页面、换文案、加组件、修构建错误，这类场景就非常适合。

但下面这些场景不适合完全依赖 Vibe Coding：

```text
支付系统
权限系统
用户隐私数据
医疗、金融、法律相关系统
大型多人协作项目
长期维护的核心业务系统
```

这些项目对安全、稳定性、可维护性的要求更高。AI 可以参与辅助开发，但不能替代严格的代码审查、测试和安全检查。

## 七、一个更稳的实践流程

如果你想用 Codex 或其他 AI 编程工具开发项目，可以按照这个流程来：

```text
1. 先说明项目背景
2. 让 AI 阅读项目结构
3. 让 AI 输出修改计划
4. 一次只做一个小功能
5. 每次修改后运行构建命令
6. 把报错原样贴给 AI
7. 功能完成后再要求重构
8. 最后补充测试和文档
```

一个比较稳的提示词可以这样写：

```text
你是我的 AI 编程助手。先不要直接写代码。
请先阅读当前项目结构，理解这个项目的技术栈。
然后根据我的需求给出修改计划，列出会修改哪些文件、为什么修改、可能有什么风险。
等我确认后，再开始改代码。
```

如果任务比较小，可以这样写：

```text
请只完成一个小任务：把首页卡片数据抽离到 src/data/sites.ts。
要求：
- 不改变页面视觉效果
- 保持 TypeScript 类型安全
- 不引入新依赖
- 修改后告诉我改了哪些文件
- 给出验证方式
```

这种方式比直接说“帮我优化项目”稳定得多。

## 总结

Vibe Coding 的本质，不是让 AI 代替开发者，而是改变开发者的工作方式。

它把编程从：

```text
我亲手写每一行代码
```

变成：

```text
我用清晰的目标和反馈，管理一个高速代码生成助手
```

所以，真正需要提升的能力也发生了变化。你不只需要懂语法，还需要懂需求表达、任务拆分、结果验证、代码审查和风险控制。

对于个人项目、静态博客、小工具来说，Vibe Coding 可以极大提高效率。但越是重要的项目，越不能只靠感觉。最稳的方式是：让 AI 快速生成，让人严格验收。

一句话概括：

> Vibe Coding 不是不写代码，而是用更高层次的方式控制代码的生成。
